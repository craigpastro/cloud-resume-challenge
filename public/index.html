The Blog Post

The other day I ran across Forrest Brazeal's Cloud Resume Challenge. I like challenges, and this one seemed quite fun, so I decided to give it a shot.

In the process of completing the challenge, I did make a few changes. These are:

  1. I didn't do Step 1. Certification. No particular reason.
  2. I also didn't do Step 6. DNS. I don't want a domain name, and I wouldn't renew it, which would mean any links to my resume would break.
  3. My resume is a link to my Linkedin. This will prevent it from getting stale.
  4. The Lambdas are in Go. No particular reason. If I had to do it again I would choose JavaScript.

Completing the challenge

I am not a cloud nooby, but I still learned a lot while completing the challenge. Let's go through the steps.

1. Certification

I didn't do this step.

2. HTML and 3. CSS

I am not an expert of HTML or CSS but I managed a little something. Emphasis on little.

4. Static Website

This was my first time deploying a website on S3. This step was more involved than uploading the files to an S3 bucket, but not complicated.

5. HTTPS

This was my first time using CloudFront. The docs at AWS were straightforward. Not using my own domain and DNS made this step a bit simpler.

6. DNS

I didn't do this step either. If I did I would have used Route 53.

7. Javascript

I'm comfortable with JavaScript, but JavaScript in the browser is a different story. I found this tutorial on How to Build Website visitor counter in JavaScript with code helpful.

8. Database

DynamoDB. Hey, something I know. Create a table, mark it as "on demand" (save money!), and off to the next step.

9. API

This was my first time using API Gateway. It was pretty straightforward to create a route and have it trigger the Lambda I wrote in the next step.

10. Python

I wrote it in Go.

11. Tests

I wrote a test.

12. Infrastructure as Code

The cloud resume challenge suggests using AWS Serverless Application Model (SAM) for this step. I suppose one could also use Terraform. I decided to try SAM since I have never used it before. After a bit of hack and deploy, I got something running. Using SAM to create the API Gateway, the DynamoDB table, upload the Lambda, and connect them all together, turned out to be easier than clicking around the console. I would still like to learn how I could do this step in Terraform.

13. Source Control

Well, GitHub of course. I created a single repository for the front and backend code. You can find it here.

14. CI/CD (Backend)

I set up two GitHub actions that are triggered by changes to the backend code. One runs backend tests on any PR that changes the backend code. The other deploys the changes to AWS when the PR is merged. The second action requires permissions on AWS, so I created an IAM user for this purpose. The hardest part of this, and the next step, was knowing which permissions to grant to this user. It was a lot of trial and error.

15. CI/CD (Frontend)

I set up a GitHub action that is triggered by changes to the frontend code. It uploads the website data to S3 and also invalidates the CloudFront distribution. This was all new to me so it was quite exciting to see it in action.
